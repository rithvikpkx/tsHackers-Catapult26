============================================================
0. PRODUCT SUMMARY
============================================================

Product name: Grind
Tagline: The AI that learns how you actually work, then rewrites your week before you fail.

Core product thesis:
Grind is not a normal todo list or planner. Grind is a behavioral correction engine for academic work. It observes how a student estimates, delays, starts, and completes tasks. It uses that behavior to predict true task duration and deadline failure risk. When a task is at risk, Grind actively intervenes by generating or moving focus blocks on the calendar and explaining why the intervention improves the odds of success.

One-sentence product definition:
Grind learns how each student personally misjudges work, predicts which deadlines are at risk, and rewrites their schedule before they fail.

============================================================
2. PRODUCT GOALS
============================================================

Grind v1 must deliver the following user-visible outcomes:
1. A student can create or ingest tasks with title, type, estimate, due date, and course.
2. Grind predicts corrected time-to-complete for each task.
3. Grind predicts failure probability for each task.
4. Grind explains why a task is risky in plain English.
5. Grind recommends or automatically creates a calendar intervention when a task is risky.
6. Grind captures enough behavioral data over time to refine per-user distortion metrics such as:
   - average start delay
   - underestimate ratio by task type
   - underestimate ratio by course
   - on-time completion rate
   - best productivity time window
7. Grind continuously updates a personalized distortion profile from user behavior.
8. Grind presents all of this in a clean frontend experience with:
   - dashboard
   - task list
   - risk alerts
   - before/after schedule comparison
   - distortion profile view

6. CORE USER EXPERIENCE FLOW
============================================================

1. User signs up and completes onboarding.
2. User enters courses and optionally imports calendar.
3. User enters tasks manually or via seeded / imported path.
4. Backend stores task and logs creation event.
5. Backend gathers current user metrics, calendar context, task metadata.
6. Backend calls ML service.
7. ML service returns:
   - predicted_actual_minutes
   - risk_probability
   - success_probability
   - explanation
   - optional recommended intervention
8. Backend stores outputs on the task row.
9. Frontend shows risk and corrected duration.
10. If risk is high, frontend shows a strong alert and the backend creates an intervention plan.
11. User starts a focus session.
12. Backend logs actual work session data.
13. User completes task.
14. Backend computes actual duration, start delay, underestimate ratio, and on-time label.
15. Rolling personalization metrics update.
16. Future predictions become more personalized.

============================================================
9. HOW TO DERIVE THE IMPORTANT BEHAVIORAL FEATURES
============================================================

9.1 Actual time to complete a task
Definition:
- sum active_seconds across all focus_sessions for the task
Formula:
- actual_minutes = sum(active_seconds) / 60

9.2 Actual start time
Definition:
- earliest focus_session.started_at for the task
Fallback:
- first task_started event

9.3 Planned start time
Definition:
- tasks.planned_start_at
Fallback:
- latest task_planned_start_set event

9.4 Start delay
Formula:
- start_delay_hours = (actual_start_ts - planned_start_ts) / 3600
Interpretation:
- positive means the user started late
- negative means the user started early

9.5 Underestimate ratio
Formula:
- underestimate_ratio = actual_minutes / estimated_minutes_initial
Interpretation:
- 2.1 means actual time was 2.1x estimate
- 0.7 means estimate was too high

9.6 On-time completion
Formula:
- completed_on_time = completed_at <= due_at

9.7 Failure label
V1 definition:
- failure_label = NOT completed_on_time

10. WEAK PRIOR MODEL STRATEGY FOR NEW USERS
============================================================

The app must work for brand-new users before enough personalization data exists.

Use a three-part initialization strategy:

10.1 Global weak prior
Train a global baseline on pooled historical task rows. This is not supposed to know the user personally. It only gives rough patterns like:
- coding assignments are often underestimated
- reading is more predictable
- dense workload weeks are riskier

10.2 Onboarding calibration
During signup, collect:
- courses this semester
- hardest course categories
- preferred work hours
- whether the user usually starts early/on time/late
- 3 to 5 recent tasks:
  - task type
  - estimated duration
  - actual duration
  - planned start
  - actual start
Use this to initialize user_metrics.

10.3 Personalization weight
Set:
- personalization_weight = min(1.0, completed_task_count / 20.0)
Meaning:
- 0 tasks -> fully weak prior
- 10 tasks -> blended
- 20+ tasks -> strongly personalized

Required blending rule:
final_prediction = (1 - personalization_weight) * global_prior_prediction + personalization_weight * personalized_prediction

============================================================
11. WHERE TO GET DATA FOR THE WEAK PRIOR MODEL
============================================================

Use a layered data strategy.

11.1 Public research-grounded baseline data
Use public education / student behavior datasets as rough priors and feature inspiration.
Recommended sources:
- OULAD (Open University Learning Analytics Dataset): student activity, assessments, course interaction data. Good for building generic risk priors and workload-related features. The repo already references an OULAD-based bootstrap prior and pipeline in builder_c, so continue using that as the current baseline source. citeturn0view0
- StudentLife dataset: smartphone-derived behavioral data tied to academic outcomes. Useful as research support for productivity-related feature families.

Important note:
Do not claim these public datasets directly produce “2.1x underestimate for OS assignments.” They do not. They only support generic priors.

11.2 Seeded historical task data from the team
For the hackathon and early production bootstrap:
- each teammate should manually enter 20 to 40 recent tasks
- include estimate, actual duration, planned start, actual start, due date, task type, course
This is enough to build the first believable personalization loop.

11.3 Additional friend-submitted data
Optional but useful:
- collect 5 to 10 tasks each from friends via a form
This improves baseline model quality.

11.4 Live product data
As the app is used, the best data source becomes the app’s own event logs and completed tasks.

13. HOW TO TRAIN THE MODELS
============================================================

13.1 Build task_features rows
Process:
- join tasks + focus_sessions + calendar context + user metrics
- derive actual_minutes, start_delay_hours, underestimate_ratio, completed_on_time, failure_label
- write one row per completed task into task_features

13.2 Training split
Use:
- time-based split if enough rows
or
- grouped split by user if enough users
For hackathon quality, a simple validation split is acceptable.

13.3 Duration model training
Steps:
1. pull rows where actual_minutes is not null
2. clean outliers
3. encode categorical features
4. train XGBoostRegressor
5. compare against naive baseline: predicted = estimated_minutes_initial
6. save model artifact and metrics
Metrics:
- MAE
- RMSE
- MAPE

13.4 Failure model training
Steps:
1. pull rows where failure_label is not null
2. include predicted_actual_minutes from duration model or cross-validated version
3. train XGBoostClassifier or LogisticRegression
4. save model artifact and metrics
Metrics:
- ROC AUC
- precision
- recall
- calibration quality if available

13.5 Global weak priors
From training data, compute global priors by task type and optionally course category.
Example output:
- coding: 1.4
- problem_set: 1.3
- writing: 1.2
- reading: 0.9
- admin: 1.1
These are fallback multipliers on user estimates.

============================================================
14. HOW TO PERSONALIZE / FINE-TUNE PER USER
============================================================

Do NOT fine-tune the whole model like an LLM.
Use residual personalization.

14.1 Global prediction
pred_global = duration_model(features)

14.2 User residual multiplier
Compute from recent completed tasks:
user_residual_multiplier = average(actual_minutes / pred_global)
Use the most recent 10 to 20 tasks.

14.3 Hierarchical fallback
For each task prediction, determine the best available multiplier from:
1. user_course_metrics if sample_count >= 5
2. user_tasktype_metrics if sample_count >= 5
3. global task_type prior
4. default global prior

14.4 Final personalized duration
pred_final = pred_global * blended_multiplier
where blended_multiplier uses personalization_weight.

14.5 Personalized failure risk
Use user-specific rolling features directly in the failure model, and optionally post-adjust risk with rules such as:
- if user avg_start_delay_hours is high and free_time_gap_minutes is negative, increase risk
- if user task-type on_time_rate is strong and task is in that type, modestly reduce risk

============================================================
15. HOW TO COMPUTE THE DISTORTION MATRIX
============================================================

Grind must compute a per-user distortion matrix that is visible in the UI.

Required computed metrics:

15.1 Global user distortion metrics
- avg_underestimate_ratio = average(actual_minutes / estimated_minutes_initial)
- avg_start_delay_hours = average(actual_start - planned_start)
- on_time_rate = completed_on_time / completed_tasks
- most_productive_hour_window = hour range with strongest historical focus completion

15.2 By task type
For each task_type:
- avg_underestimate_ratio_task_type
- avg_start_delay_hours_task_type
- on_time_rate_task_type
- avg_actual_minutes_task_type

15.3 By course
For each course:
- avg_underestimate_ratio_course
- avg_start_delay_hours_course
- on_time_rate_course
- avg_actual_minutes_course

Display examples:
- “You underestimate systems assignments by 2.1x.”
- “You start problem sets an average of 1.8 days later than planned.”
- “You are most productive between 9 PM and 1 AM.”

============================================================
16. HOW TO DETECT BEST PRODUCTIVITY TIME RANGE
============================================================

Build this from completed focus session history.

Method:
1. bucket all focus session active_seconds by hour-of-day
2. optionally weight by session completion quality or whether the related task finished on time
3. compute rolling average active productive minutes per hour bucket
4. find top contiguous 2 to 4 hour window
Store in user_metrics:
- most_productive_hour_start
- most_productive_hour_end

Use this window to prefer intervention placement.

============================================================
17. INFERENCE AND ALERTING PIPELINE
============================================================

17.1 When a task is created or updated
Backend should:
1. gather task fields
2. gather user metrics
3. gather task-type and course metrics
4. gather calendar context and free-time features
5. call ML service /predict-task
6. store outputs on task row
7. if risk > threshold, generate intervention candidate

17.2 Risk threshold
Default threshold:
- risk_probability > 0.65
or
- free_time_gap_minutes < 0
Then:
- create an intervention
- show alert
- optionally auto-place focus block if user has enabled it

17.3 Big-picture AI model / orchestration layer
There should be a higher-level orchestration component that looks at:
- ML outputs
- current calendar state
- user metrics
- active tasks
- upcoming deadlines
and decides when to send notifications or create interventions.

This can be implemented as a backend orchestration service, not necessarily a separate giant model.

Responsibilities:
- prioritize which risky task to warn about
- avoid spamming the user
- decide whether to recommend immediate action or calendar intervention
- generate a clear explanation string
- send notification through frontend and optionally email/push later

Implement this as rule-driven orchestration first, with optional LLM-generated explanations.
Do NOT build an overcomplicated autonomous agent for v1.

============================================================
18. INTERVENTION ENGINE REQUIREMENTS
============================================================

When a task is risky, Grind should create a concrete plan.

Preferred intervention order:
1. create a focus block in the best free window
2. move a flexible Grind-generated block
3. move a flexible external block
4. split a task into two or more focus blocks
5. recommend immediate start

Each intervention must store:
- risk_probability_before
- risk_probability_after
- freed_minutes
- explanation
- exact changed events in action_json
